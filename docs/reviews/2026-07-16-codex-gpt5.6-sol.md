# Architecture review — Codex (gpt-5.6-sol)

Date: 2026-07-16. Read-only architecture review of `docs/architecture.md` and the implementation,
run via `codex exec -m gpt-5.6-sol -s read-only`. File references were de-absolutized to `file:line`.
Tracked as GitHub issues (see the repo). Some section-1 doc/comment nits were fixed in the pre-publish cleanup.

## 1. Correctness: documentation versus implementation

Overall, `docs/architecture.md` accurately describes the current demo-level flow, but several claims are imprecise or stronger than the implementation warrants.

### Inaccuracies and drift

- Callback renames are not fully tracked automatically. The document says renaming `dispatch` or the `app` package path is handled automatically (`architecture.md:129`). In fact, extraction depends on the hard-coded suffix `3app8dispatch` (`build.sh:19`). Renaming either requires manually updating `PKG_FN_SUFFIX`; otherwise extraction fails. Toolchain changes are only handled if the new symbol still matches the existing regex structure (`build.sh:27`).

- The statement that `build.rs` generates the callback declaration “from the real mangled symbol, extracted at build time” is directionally correct but conflates two builds. `build.sh` performs extraction (`build.sh:23`); `build.rs` merely consumes `mb_symbol.txt` (`build.rs:10`). A bare Cargo build can consume an old, valid-looking file.

- “The root gets `track_focus` and `on_key_down`” is misleading (`architecture.md:37`). Those handlers are attached to a Rust-created outer container around all live top-level nodes (`lib.rs:211`), not to the MoonBit-created root node.

- `NODES` is described as holding “the tree” (`architecture.md:32`). More precisely, it is an arena of construction slots. `gpui_add_child` removes a node from its slot with `take()` and embeds it by value under its parent (`lib.rs:169`). Only unattached `Some` slots are rendered as roots (`lib.rs:213`). This distinction matters for handle validity.

- The documentation calls handles indices into `NODES` and notes move semantics, but does not state that a child handle becomes invalid after `add_child`. MoonBit’s `NodeHandle` remains an unrestricted alias for `Int` (`gpui-bindings.mbt:4`), so the API does not encode this ownership transition.

- “Adding event types does not grow the FFI” is literally true for the number of symbols (`architecture.md:71`), but not for the contract. Every new kind still changes a manually synchronized, unversioned wire protocol across Rust and MoonBit (`lib.rs:21`, `app.mbt:22`).

- The app comment says the key event is reserved and not emitted by Rust (`app.mbt:24`); Rust currently emits it (`lib.rs:227`). This is direct comment drift.

- The diagram says MoonBit main calls `build_tree`, whose first operation is reset (`architecture.md:82`). That is correct, but the text elsewhere can read as if main separately calls `gpui_reset`; it does not (`main.mbt:7`, `app.mbt:99`).

- “`gpui_reset()` clears `NODES` … done on every event” is true even for unknown or ineffective events: `dispatch` always rebuilds after decoding (`app.mbt:43`). The documentation implies state-changing events, but malformed/unknown events incur the same rebuild.

- The build documentation says the first MoonBit link failure is ignored on a cold build (`architecture.md:115`). The script actually ignores every failure from the first `moon build`, including compiler/type errors (`build.sh:23`). Subsequent symbol extraction may then use stale object files.

- The project is characterized as macOS/arm64 and Mach-O, which matches the scripts, but the build configuration embeds absolute machine-specific paths (`moon.pkg:10`). The architecture document should explicitly call the current build non-relocatable.

## 2. Design soundness

### Retained tree and integer handles

What is good:

- The model cleanly isolates MoonBit from GPUI’s Rust types, lifetimes, generics, closures, and entity/context APIs.
- Building an inert description in MoonBit and translating it during GPUI render is a sensible bridge for a language with limited callback and native-ABI support.
- Move-on-attach gives Rust sole ownership of nested nodes and avoids cross-language object lifetime management.
- Rebuilding eliminates stale-property and incremental reconciliation bugs for this small demo.

What is questionable:

- This is better described as a transient retained description than a conventional retained UI tree. The arena is both a builder workspace and a root registry, while attached nodes leave the arena.
- Raw `i32` handles provide no generation, type, tree, or ownership information. A handle can be negative, out of range, already moved, or refer to the wrong variant.
- Error handling is inconsistent. Most setters silently ignore bad handles or wrong node types (`lib.rs:77`), whereas `gpui_add_child` directly indexes and can panic (`lib.rs:172`). Worse, it takes the child before validating the parent, so a valid child is silently lost if the parent is invalid or is text.
- `NODES` is process-global (`lib.rs:6`). This precludes independent windows/views and makes ownership implicit.
- Every tiny setter acquires the global mutex separately. The mutex adds little safety under the stated single-main-thread restriction, while creating lock overhead and poisoning risk.

### Single scalar callback

What is good:

- One callback is pragmatic given MoonBit’s lack of a stable executable export and C-compatible closure facility.
- Scalar values avoid retaining MoonBit heap objects in Rust, consistent with the native runtime constraints (`moonbit-native-notes.md:87`).
- A central decoder is easier to audit than many separately mangled callback declarations.

What is questionable:

- `(kind, id, a, b)` is an undocumented wire ABI disguised as four integers. Field meanings, modifier bits, signedness, units, and versioning are implicit.
- It fits clicks and simple key codes, but not text input, IME composition, clipboard data, paths, accessibility actions, rich pointer events, or payloads with floating-point coordinates without ad hoc packing or side channels.
- Constants are duplicated manually. A mismatch remains valid at both ABI and link levels and can silently route an event incorrectly.
- The callback’s Rust declaration assumes MoonBit `Int` is ABI-compatible with `i32`. The background document records this experimentally, but there is no automated ABI check.
- MoonBit panic behavior is process-abort and the runtime is non-atomic RC (`moonbit-native-notes.md:75`); therefore every callback must remain main-thread-only and effectively total. The Rust API does not enforce that architectural requirement.

### MoonBit-owned main and Rust static library

This is sound under the documented MoonBit-native constraints. It preserves MoonBit runtime initialization and avoids competing `_main` symbols (`moonbit-native-notes.md:69`). A static library also allows the callback reference to remain unresolved until the final MoonBit executable link.

The cost is substantial operational coupling:

- Rust’s native dependencies become manually maintained linker configuration.
- The library cannot be independently linked or tested as a complete artifact without a callback provider.
- Cargo and Moon use separate dependency graphs, so the custom script must simulate missing cross-build dependencies.
- The design is currently tied to Mach-O symbol decoration and specific MoonBit output directories.

Given the toolchain limitations, it is a reasonable prototype architecture, but it should be treated as a platform-specific integration adapter, not yet a stable general-purpose binding architecture.

## 3. Risks, fragility, and footguns

### Mangled-symbol extraction

Only partially adequate.

It removes the need to paste the full mangled symbol, but still hard-codes the mangled suffix and Mach-O underscore convention (`build.sh:21`, `build.sh:36`). It also searches every object below `_build/native`, sorts matches, and takes the first (`build.sh:27`). Stale builds or multiple configurations can select the wrong object.

Most seriously, ignoring the initial build failure allows stale-symbol extraction after a genuine MoonBit compile failure. This can mask the real error until the final build or, if stale objects remain ABI-compatible enough, select unintended output.

`mb_symbol.txt` also contains only a symbol, with no MoonBit version, target, package identity, signature hash, or build-output provenance. Cargo can therefore accept a stale symbol as current.

### Forced relink

Adequate for the exact current debug layout, fragile beyond it.

Removing `main.exe` and one linked-core object forces the observed Moon version to relink (`build.sh:47`). It assumes fixed debug paths and Moon’s current internal directory structure. Release builds, renamed commands, multi-target builds, or a Moon layout change can bypass it.

The script also deletes derived outputs before the final build. That is acceptable in a build driver, but a more robust integration would declare the archive as an actual dependency or use a generated stamp consumed by Moon’s build graph.

### DCE retention

The explicit reference is a practical mitigation (`main.mbt:2`), but it relies on optimizer behavior: `_keep` is immediately passed to `ignore`. There is no post-build assertion that the callback symbol survived. A toolchain optimizer could eventually prove the value unused.

A build-time `nm` assertion on the final relevant object/executable is needed. Current extraction incidentally requires the symbol before the final build, but does not verify the final linked executable afterward.

### `.app` keyboard workaround

Adequate for local macOS use, not a general input architecture.

The bundle provides identity and the app metadata reported to fix keyboard delivery (`bundle.sh:24`). However:

- `build.sh` does not invoke `bundle.sh`.
- There is no signing, hardened-runtime, entitlement, icon, or release packaging flow.
- The claimed minimum macOS version is asserted in the plist but not aligned with a Rust deployment target (`bundle.sh:42`).
- Key handling only accepts a one-character GPUI key name; named keys and multi-character names are discarded (`lib.rs:240`). This is not suitable for general keyboard or text input.

### Manual framework list

This is the largest build-maintenance footgun. The list is duplicated knowledge from Cargo and includes an extra/manual ordering choice (`moon.pkg:11`). It can silently remain sufficient while omitting a newly conditional dependency, then fail only on a clean machine or different feature set.

The recommended Cargo command is useful, but there is no automated comparison or regeneration. Absolute `-I` and `-L` paths make the project immediately non-portable to another checkout.

### Full clear and rebuild

Adequate for a counter; not adequate as an unqualified long-term mitigation.

Benefits are simplicity and deterministic state. Remaining issues include:

- O(number of nodes) allocations and many FFI calls on every handled, ignored, or unknown event.
- A mutex lock/unlock for nearly every node property and attachment.
- All text strings are re-encoded and copied MoonBit → Rust on each rebuild.
- The tree is mutated incrementally in the live global store. There is no atomic commit, so a panic or future asynchronous render could expose an empty or partially built UI.
- Stateful GPUI element identity is synthesized from `click_id` (`lib.rs:309`). Duplicate click IDs create duplicate element IDs, and non-clickable stateful elements have no stable identity scheme.
- Multiple windows cannot retain independent trees.

### Additional silent-break cases

- Negative or invalid handles can become huge `usize` values. Most operations silently do nothing; `add_child` panics.
- Attaching the same child twice silently does nothing on the second call.
- Attaching to a non-div parent removes and loses the child.
- Cycles are prevented incidentally by move semantics, but reparenting an already attached node is impossible.
- Embedded NUL in a MoonBit string truncates the Rust text; invalid UTF-8 becomes the empty string without error (`lib.rs:149`).
- A Rust panic across an `extern "C"` export can abort the process; mutex poisoning then makes subsequent accesses panic as well.
- The final `nm`/ABI/link outcome is not smoke-tested by launching the application or synthesizing an event.

## 4. Scalability and maintainability

The current architecture scales acceptably in number of buttons, but poorly in UI expressiveness, event richness, and number of views.

Adding styles currently requires synchronized changes across:

1. `UiNode`,
2. a Rust C export,
3. cbindgen output,
4. generated MoonBit declarations,
5. the MoonBit wrapper,
6. render translation.

This produces a wide, function-per-property ABI. It will become cumbersome for margins, padding, borders, overflow, positioning, typography, images, lists, scrolling, and accessibility. Boolean one-way setters such as `set_center` and `set_flex` also make reuse or mutation awkward.

The event ABI avoids symbol proliferation but moves complexity into integer schemas. Once payloads exceed three scalar fields, likely outcomes are overloaded meanings, bit packing, global side buffers, or additional getter calls. Text input alone requires a length-aware byte payload and explicit lifetime rules.

The ownership model becomes difficult for reusable components because a `NodeHandle` is consumed operationally but not in MoonBit’s type system. Component code cannot safely retain or share handles. There is also no explicit root handle; every unattached node is rendered, so a forgotten attachment can accidentally create another root.

Performance will degrade with larger trees because each render reconstructs GPUI elements recursively and each event reconstructs the Rust description through many independently locked C calls. For animation, pointer movement, text editing, or scrolling, full rebuild on every event will likely be prohibitive.

Maintainability is also limited by the absence of automated contract tests. The most dangerous contracts—event constants, callback signature, native symbol presence, framework flags, and handle validity—are expressed in comments and scripts rather than checked artifacts.

## 5. Prioritized improvements

### P0 — Prevent ABI mismatches and stale/wrong callback selection

Generate a shared event/ABI definition containing:

- ABI version,
- callback arity and fixed-width types,
- event kind values,
- modifier-bit definitions.

Use it to generate both the Rust declarations/constants and MoonBit constants. After the final link, verify with `nm` that exactly one expected callback definition exists.

Also make symbol extraction target the exact current app object rather than all `_build/native` objects, and do not broadly ignore the first MoonBit build failure. If cold-start failure genuinely must be tolerated, distinguish “link failed solely because `gpui_sys`/callback is absent” from compilation failures.

Rationale: signature mismatch is silent UB; wrong constants are silent semantic corruption; stale extraction can hide build errors.

Effort: medium, roughly 1–3 days depending on MoonBit code-generation integration.

### P0 — Make all C exports panic-safe and validate handles consistently

Replace raw indexing in `gpui_add_child` with checked access and validate both parent and child before consuming either. Define consistent status-returning APIs, for example `int32_t` error codes, for every operation. Reject negative handles, moved handles, wrong variants, duplicate attachment, and overflow.

Avoid letting mutex poisoning or Rust panics cross the C boundary. At minimum, make invalid caller input non-panicking; preferably wrap exported entry points with an abort-safe/error-reporting boundary appropriate to the project’s panic strategy.

Rationale: current malformed handles can either silently corrupt the constructed tree or terminate the process.

Effort: small to medium, about 1 day plus tests.

### P0 — Remove absolute build paths

Generate link configuration from the checkout root or have `build.sh` pass paths into the Moon build. At minimum, generate the relevant `moon.pkg` fragment deterministically rather than committing `/Users/...` paths.

Rationale: the current project cannot reliably build from another directory or machine.

Effort: small, hours to 1 day.

### P1 — Introduce an explicit builder transaction and root

Use an API resembling:

- `begin_tree(view_id)`
- create/configure nodes in a staging arena
- `set_root(builder, handle)`
- `commit_tree(builder)`

Only swap the committed tree once construction succeeds. Scope storage by view/window rather than one global `NODES`.

Rationale: provides atomic rebuilds, makes the root explicit, enables multiple windows, and creates a place to validate the entire tree before rendering.

Effort: medium, 2–4 days.

### P1 — Replace property-per-call construction with batched node descriptions

Expose constructors accepting a C-compatible style struct, or serialize a compact length-delimited command buffer from MoonBit. A command-buffer approach can build the complete tree with one or a few FFI calls.

Rationale: reduces lock crossings, FFI boilerplate, generated declarations, and per-property synchronization work. It also makes adding style fields more manageable through versioned structures or opcodes.

Effort: medium to large, approximately 3–7 days.

### P1 — Evolve the callback into a versioned event envelope

Keep one callback, but define an envelope such as `(abi_version, event_kind, payload_ptr, payload_len)` with payload memory valid only during the call. Retain a compact fixed struct for common scalar events and support bytes for text/IME data.

If MoonBit’s pointer/borrow facilities make that unsafe, use an event queue owned by Rust plus scalar `event_token`, with MoonBit calling typed getter/copy APIs synchronously.

Rationale: preserves the single callback while supporting text input and richer events without ad hoc packing.

Effort: medium, 2–5 days.

### P1 — Automate native library flag verification

Have the build query `cargo rustc ... --print native-static-libs`, normalize its output, and either generate the Moon link flags or fail when the committed configuration differs. Use the same Cargo profile as the archive linked by Moon.

Rationale: turns an upgrade-time checklist into a reproducible build invariant.

Effort: small to medium, 1–2 days.

### P1 — Add cross-boundary integration tests

At minimum test:

- valid and invalid handle operations,
- move-on-attach behavior,
- duplicate/wrong-parent attachment,
- event constant compatibility,
- final callback symbol presence,
- non-ASCII and embedded-NUL text behavior,
- clean build from an empty `_build`/`target`,
- rebuild after a Rust-only source change.

Rationale: most architectural risks live between compilers and are not covered by unit tests in either language alone.

Effort: medium, 2–4 days.

### P2 — Add stable identity independent of click IDs

Give nodes explicit stable keys supplied by MoonBit. Event IDs and GPUI element identities should be separate concepts, with duplicate-key validation.

Rationale: click IDs are action-routing data, not universally valid retained-element identities.

Effort: small to medium, 1–2 days.

### P2 — Add incremental updates only where measurements justify them

After batching and transactional commits, profile realistic trees. Then consider keyed subtree replacement or direct state updates for high-frequency data such as text editing and pointer movement. Do not begin with a general virtual-DOM diff unless needed.

Rationale: full rebuilding is an excellent simplicity baseline, but high-frequency events need a path that avoids rebuilding unrelated nodes.

Effort: medium to large, depending on scope.

### P2 — Formalize platform and toolchain support

Pin or validate MoonBit, Rust, GPUI, target architecture, deployment target, and Mach-O assumptions. Make `build.sh` fail early with actionable messages on unsupported configurations. Document that the bundle flow is development-only unless signing/release packaging is added.

Rationale: the design depends on observed, unstable compiler and linker behavior (`moonbit-native-notes.md:6`).

Effort: small, about 1 day.

The central architectural choice—MoonBit-owned main, Rust static library, transient retained description, and one scalar callback—is defensible for the current toolchain and counter demo. The immediate priority is not replacing it, but turning its implicit ABI, handle ownership, and build coupling into checked contracts.
